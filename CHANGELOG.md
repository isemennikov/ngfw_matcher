# CHANGELOG

## [2.4.0] — 2026-07-14

### Добавлено

- **Прогресс-бар для Shadows** — O(n²) расчёт (10–40 сек на больших наборах правил)
  теперь выполняется в фоновом потоке с job-based polling (тот же паттерн, что и
  у Sync), а не синхронно в одном HTTP-запросе. `core/matcher.py`: `check_shadowed`/
  `check_partial_shadowed` принимают `progress_cb(current, total)`, вызываемый
  каждые ~1% внешнего цикла. Прогресс хранится in-memory (`web/state.py`,
  без БД — результат нужен только на время сессии), эндпоинты `POST /shadows` +
  `GET /shadows/status/{job_id}` в `web/routers/shadows.py`. Постраничная навигация
  по готовому результату (`/shadows/status/{job_id}?page=N`) не пересчитывает анализ.

- **Показать/скрыть пароль** на `/connect` — иконка-глаз в поле пароля.

- **Favicon, `theme-color`, `meta description`** — раньше вкладка браузера была
  пустой, страница не описана для превью/шаринга.

### Изменено

- **Tailwind CDN → статическая сборка** — `cdn.tailwindcss.com` (полный JIT-движок
  в браузере на каждый заход) заменён на собранный `web/static/css/tailwind.css`.
  Добавлен `package.json` + `tailwind.config.js`, сборка — отдельный Node-стейдж
  в `Dockerfile` (`npx tailwindcss --minify`), копируется в финальный образ.

- **Базовый размер шрифта** — `text-xs` 12→13px, `text-sm` 14→15px через
  `tailwind.config.js` (`theme.extend.fontSize`), без правки каждого из 20 шаблонов.
  Интерфейс рассчитан на многочасовые сессии анализа — мелкий шрифт напрягал глаза.

- **Max-width на контент** — `layout.html` оборачивает `{% block content %}` в
  `max-w-6xl`, чтобы на широких мониторах строки таблиц и карточек не растягивались
  на всю ширину.

- **Sync без перезагрузки страницы** — дерево устройств и статус-бокс выделены в
  партиалы `partials/device_tree_rows.html` / `partials/device_status_box.html`,
  подключаются через `{% include %}` в сайдбаре и обновляются HTMX out-of-band
  swap'ом (`hx-swap-oob`) после завершения sync — вместо `window.location.reload()`.

- **Hits: сортировка через Alpine.js** — переключатель «по хитам / по порядку
  политики» больше не зависит от скрытого поля + делегированного JS-обработчика
  на `document`, состояние — реактивное `x-data` на контейнере страницы.

### Улучшено (UX)

- **Доступность** — `aria-label` на все иконки-кнопки в сайдбаре (sync, upload,
  download), `aria-hidden="true"` на декоративные SVG и thread-dots, ссылка
  «Перейти к содержимому» (skip-to-content), `:focus-visible` outline для
  клавиатурной навигации.

- **Тултип для «глобальная позиция»** — в Match / Find Rule поясняет, что это
  порядковый номер в объединённом списке PRE+POST (с учётом родительских групп,
  если включено наследование).

## [2.3.1] — 2026-07-13

### Изменено

- **Один Sync на всю СУ** — кнопка ↻ убрана с каждой строки дерева устройств и
  перенесена в заголовок секции «Устройства» (одна кнопка на всё дерево). Ранее
  синхронизация запускалась по клику на конкретное устройство, но фактически всегда
  выгружала всё дерево групп целиком (`GetDeviceGroupsTree`) — просто сохраняла
  результат в отдельный файл на каждое устройство, из-за чего в сайдбаре дублировались
  два независимых списка («Устройства» и «Группа устройств»), указывающие по сути
  на одни и те же группы. Теперь один sync сохраняет единый снапшот на хост, все узлы
  дерева получают актуальные `snapshot_path`/`rules_count` за один вызов, а выбор узла
  в дереве сам определяет активную группу правил — отдельный селектор «Группа устройств»
  показывается только для вручную загруженного оффлайн multi-снапшота.

### Исправлено

- **Rule Hits могли синхронизироваться не под той группой** — если выбранное устройство
  не совпадало с активной группой правил (`active_group_id` по умолчанию брался из
  первой группы снапшота), счётчики hits сохранялись под неверным `device_id`. После
  унификации выбора устройства и активной группы правил это стало невозможно.

## [2.3.0] — 2026-06-29

### Добавлено

- **Сравнение правил в Shadows** — рядом с каждым теневым правилом появилась кнопка
  **⇄ сравнить**. При клике HTMX-запросом рендерится инлайн-панель с side-by-side
  сравнением пары: Source / Destination / Service.

  Каждый объект подсвечивается по статусу пересечения:
  - 🟢 **полное** — один объект содержит другой или равен ему
  - 🟡 **частичное** — диапазоны адресов или портов частично перекрываются
  - ⚫ **нет** — пересечения не найдено

  Реализовано через новый модуль `core/rule_compare.py` (`compare_rules`,
  `RuleCmp`, `FieldCmp`, `ObjItem`) и роутер `web/routers/compare.py` (`POST /compare`).
  Каждый объект резолвится индивидуально через существующий `ObjectResolver`.

### Исправлено

- **RFC1918 фильтр для SNAT ANY** — SNAT-правила с `destination=ANY` больше не
  ассоциируются с правилами безопасности, у которых назначение — приватные адреса
  (RFC1918). В окружении заказчика `ANY` в SNAT означает только интернет-трафик,
  поэтому ассоциации вида SNAT ANY → security rule с dst `10.x/24` были ложными.

## [2.2.0] — 2026-06-29

### Добавлено

- **Глобальный Sync всех групп** — кнопка ↻ теперь выгружает всё дерево групп
  устройств за один раз через `GetDeviceGroupsTree`. Все группы загружаются
  параллельно (`asyncio.gather`). Снапшот сохраняется в едином файле в формате
  `{"device_groups": {"<gid>": {rules, nat_rules, net_groups, svc_groups}}}`.
  Обратная совместимость с legacy-форматом сохранена через `core/snapshot.py`.

- **Селектор группы устройств** в sidebar — после Sync отображается дерево групп
  с отступами, счётчиком правил и подсветкой активной группы. Переключение группы
  меняет контекст анализа без повторного Sync.

- **`nat-audit`** — новая команда и веб-страница для просмотра NAT правил PT NGFW.
  Каждое правило отображается с типом (`SNAT` / `DNAT` / `SNAT+DNAT`),
  направлением (`-->` / `<--` / `<-->`), условиями матчинга (src, dst, svc)
  и адресами трансляции. Фильтрация по типу через `--type snat|dnat|all`.

- **NAT ↔ security rule association** — для каждого NAT правила находятся
  правила безопасности с пересекающимися адресами/сервисами. Для DNAT используется
  `dstTranslatedAddress` (PT NGFW применяет политику безопасности после трансляции).
  Зональные правила и правила с FQDN-объектами исключаются во избежание ложных
  ассоциаций. Требуется нетривиальное пересечение (хотя бы по одному измерению).

- **JSON export для NAT audit** — CLI: `--json FILE|-`; веб: кнопка экспорта на
  странице NAT Audit. Каждое правило содержит поля NAT и список ассоциированных
  правил безопасности с `full_coverage`, `conflict_dims`, `has_conflict`.

- **Легенда NAT Audit** — описание значков (`ALLOW`/`DENY` badge, `✓ full`,
  `⚠ partial`) и индикаторов покрытия на странице веб-интерфейса.

- **HTMX и Alpine.js раздаются локально** — JS-библиотеки бандлируются в образ
  (`web/static/js/`), устранена зависимость от внешних CDN.

- **Автообновление страницы после Sync** — после завершения синхронизации страница
  перезагружается автоматически: sidebar сразу показывает актуальные дату и
  количество правил без ручного F5.

- **Глобальная полоска загрузки** — синяя линия фиксируется вверху страницы при
  любом HTMX-запросе (через события `htmx:beforeRequest` / `htmx:afterRequest`).

### Улучшено (UX)

- **Ошибка Sync** — полный текст исключения виден сразу (до 120 символов),
  полное сообщение доступно через `title` при наведении.

- **Empty state** — страница "нет данных" теперь контекстная: три состояния —
  устройство не выбрано (пошаговая инструкция), выбрано но Sync не выполнен
  (мигающий индикатор и подсказка), снапшот не найден (ссылка на загрузку).

- **Статус Sync в sidebar** — компактная строка `✓ Снапшот актуален · дата`
  с числом правил; мигающая янтарная точка когда Sync не выполнен.

### Исправлено

- **NAT ассоциации в мультигрупп снапшоте** — веб-роутер читал `rules`,
  `net_groups`, `svc_groups` из корня снапшота (там пусто в новом формате).
  Исправлено через `get_group()` + `get_active_group_id()`.

## [1.5.1] — ветка fix_check-shadowed

### Исправлено

- **`check-shadowed`**: ложные теневые правила из-за FQDN и Application-only правил.

  `resolve_field_network()` возвращает `ANY_NET` (`0.0.0.0/0`) для FQDN-объектов
  как консервативный fallback — это корректно для `match`, но в shadow-анализе
  делало такое правило «покрывающим всё», порождая ложные пары.
  Аналогично, app-only правила (`service=ANY` + `application=LIST`) резолвятся
  в `ANY_SVC`, что давало ложное покрытие по портам.

  Оба типа теперь исключаются из анализа — так же как зональные правила.
  Строка статистики в выводе дополнена счётчиком пропущенных FQDN/app-only:
  `зональных: N  FQDN/app-only: M  в анализе: K`

## [1.5.0] — ветка feature/rule-hits

### Добавлено

- **`rule-hits`** — новая команда: счётчики срабатываний правил (`hits`) через
  `POST /api/v2/ListMetricsRulesStats`. Запрашивает API батчами по 30 правил
  (настраивается через `--batch-size N`).

  Вывод: таблица с именем правила, счётчиком в формате `1.2K` / `3.4M` / `1.5B`
  и 22-символьной ANSI-шкалой. Цвет шкалы — по перцентилям от общего распределения:
  зелёный (нижняя треть) → жёлтый (средняя) → красный (верхняя треть), серый — 0 hits.
  Выключенные правила отображаются приглушённо.

  Флаги команды:
  - `--device ID` — deviceGroupId (иначе интерактивный выбор, как в остальных командах)
  - `--rule PATTERN` — фильтр по имени (подстрока, регистронезависимо)
  - `--sort-hits` — сортировать по убыванию hits; без флага — порядок как в СУ
  - `--batch-size N` — размер батча (по умолчанию: 30)

### Исправлено

- **`rule-hits`**: дублирование строк в выводе из-за того, что `get_rules()` возвращает
  три набора (PRE + default + POST), где `default` дублирует PRE/POST.
  Фикс: фильтрация `_precedence == "default"` + дедупликация по rule ID
  (аналогично тому, как это делает `build_matcher`).

## [Unreleased] — ветка dev_new_features_2025

### Добавлено

- **`find-rule PATTERN`** — новая команда для поиска правил по подстроке имени или UUID.
  Выводит карточку правила с раскрытием всех полей (source, destination, service, application).
  Группы разворачиваются через API с усечением до 10 объектов.
  Поддерживает `--output FILE.json` для экспорта найденных правил.

- **`check-shadowed`** — новая команда математического анализа теневых правил.
  Для каждой пары правил (A выше B) проверяет условие полного покрытия:
  `A.src ⊇ B.src` И `A.dst ⊇ B.dst` И `A.svc ⊇ B.svc`.
  Если выполнено — правило B гарантированно теневое.
  Особо выделяет конфликты действий (например, ALLOW перекрывает явный DENY).
  Поддерживает `--output FILE.json`.

- **`match --fullview`** (расширение) — теперь работает без `--output`.
  При указании `--fullview` в терминальном выводе matched/shadowed правил
  группы раскрываются с усечением до 5 объектов + «+N ещё».
  При совместном использовании `--fullview --output FILE.json` дополнительно
  сохраняет JSON-скан всех правил, совпадающих по src.

### Изменено

- **`match`**: удалены неиспользуемые аргументы `--sport`, `--zone-src`, `--zone-dst`.
  Анализ выполняется только по IP/CIDR, порту и протоколу — зональные правила
  пропускаются на уровне матчера.

## [1.0.0] — первый релиз

- Симуляция трафика по правилам PT NGFW (first-match семантика).
- Два источника данных: прямой API (`--source ngfw`) и кэш ngfw-manager (`--source backend`).
- Режимы: одиночный запрос, batch CSV, интерактивный.
- Определение теневых / дублирующих правил.
- Экспорт результатов в CSV (`--output`).
- Оффлайн-режим (`--rules-file`).

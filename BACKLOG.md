
1. **object-usage — где используется объект**
Принимает имя или IP-сеть, возвращает все правила которые на него ссылаются. Нужно перед тем как удалять или переименовывать сетевой объект.

2. **coverage — аудит покрытия хостов**
Принимает список IP/CIDR (например VLAN или сегмент), показывает для каких из них нет ни одного ALLOW-правила. Ответ на вопрос «все ли сервера в этой подсети имеют разрешение на выход?»

3. **snapshot — офлайн-кэш состояния политики**  **DONE***
   Команда скачивает правила + все объекты/группы в один файл. Остальные команды работают с `--snapshot FILE` без обращений к API.


4. **scan-source — отдельная команда вместо match --fullview**
Вынести `--fullview` из `match` в самостоятельную команду `fullview` (или пару `scan-source` / `scan-destination`).
Текущий `match --fullview` концептуально не про симуляцию трафика, а про поиск правил по адресу.

Дизайн:
- `ngfw-match fullview --src 20.20.20.20/32 --output rules.json`
  → все правила где sourceAddr покрывает указанный адрес
- `ngfw-match fullview --dst 20.20.20.0/24 --output rules.json`
  → все правила где destinationAddr покрывает указанный адрес
- Флаг `--overlap` переиспользует уже готовый механизм overlap из `match`:
  с `--overlap` и `--src 20.20.20.0/24` покажет также правила с более мелкими
  сетями/хостами внутри этой подсети (не только суперсети).
- `--output FILE.json` — экспорт с раскрытыми объектами (как сейчас в fullview_scan).
